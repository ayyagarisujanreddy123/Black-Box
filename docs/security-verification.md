# Security verification map

> Source candidate: Black Box 0.1.0
>
> Verified: 2026-07-21

This document maps every item in the execution plan's privacy and security checklist to the implementation boundary and regression evidence that enforce it. It is intended to make the security posture reviewable instead of relying on an unchecked claim.

This is not a third-party security audit, a sandbox guarantee, or a claim that captured evidence is harmless. The trust boundaries and vulnerability-reporting process remain defined in the [security policy](../SECURITY.md).

## Verification commands

The complete local source gate runs formatting, lint, strict typechecking, unit tests, and the packaged end-to-end path:

```bash
npm run check
```

The release candidate also runs the package smoke test, benchmark, two offline demo rehearsals, and dependency audit described in the [release checklist](release-checklist.md). Platform CI must be associated with the exact candidate commit before a public release claim is made.

## Residual risks

These controls reduce specific failure modes; they do not eliminate the product's inherent sensitivity:

- request bodies, source files, terminal output, and model text can contain secrets that header filtering cannot recognize;
- rule-based share redaction can miss novel credentials, personal data, or sensitive prose;
- anyone who can read the Black Box home may be able to read captured evidence;
- a forensic archive intentionally contains full retained evidence;
- the wrapped agent is not sandboxed and Black Box does not prevent its filesystem or network actions;
- archive hashes detect modification but are not signatures;
- loopback services still depend on host and browser security;
- no WebSocket/Realtime capture or public custom adapter-event ingestion exists in 0.1.

Review exported artifacts before sharing them, keep the Black Box home private, and report suspected vulnerabilities through the private process in [`SECURITY.md`](../SECURITY.md).
